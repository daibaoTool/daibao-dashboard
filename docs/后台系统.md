# CoWatch 后台系统设计

## 一、整体定位

CoWatch 体系由多个独立仓库组成，各自职责清晰：

| 仓库 | 职责 | 类型 |
|------|------|------|
| `monitor` | 前端监控 SDK（npm 包） | 数据采集 |
| `monitor-backend` | 接收 SDK 上报，写入监控数据库 | 写入侧 |
| `CoWatch` | CoWatch 业务前端 | 产品 |
| `CoWatch-backend` | CoWatch 业务逻辑 + Admin 接口 | 写入侧 + 管理侧 |
| `cowatch-admin`（待建） | 大盘网前端（Monitor / Business / Admin 三模块） | 消费侧 |

**核心原则：monitor-backend 和 CoWatch-backend 都是写入侧，cowatch-admin 是唯一的读取/管理消费端。**

## 二、后台系统（cowatch-admin）设计

### 2.1 三模块划分

大盘网包含三个功能模块，共用一套登录体系和权限系统。三模块结构完全对称，**全部带 `/:appKey` 前缀**：

```
cowatch-admin
  ├── Monitor 模块   /:appKey/monitor/*   只读，数据来源：monitor-backend
  ├── Business 模块  /:appKey/business/*  只读，数据来源：CoWatch-backend
  └── Admin 模块     /:appKey/admin/*     读写，数据来源：CoWatch-backend
```

**Monitor**：前端性能指标（FCP/LCP/CLS/TTFB/INP）、错误追踪、PV/UV 趋势。

**Business**：房间统计、用户增长、流量/成本分析。内测阶段用于分析实际使用成本，指导定价策略。

**Admin**：用户管理（封号/解封、手动设置会员等级、删除用户）、房间管理、邀请码管理。测试环境下无需直接操作数据库。

### 2.2 路由结构

```
/login                            → 统一登录

/cowatch/monitor/perf             ┐
/cowatch/monitor/errors           ├─ Monitor 模块（appKey = cowatch）
/cowatch/monitor/pv               ┘

/cowatch/business/rooms           ┐
/cowatch/business/users           ├─ Business 模块（appKey = cowatch）
/cowatch/business/traffic         ┘

/cowatch/admin/users              ┐
/cowatch/admin/rooms              ├─ Admin 模块（appKey = cowatch）
/cowatch/admin/invite-codes       ┘
```

用户在左侧点击产品（appKey），右侧呈现 Monitor / Business / Admin 三个 tab，三者结构完全对称，心智模型统一。

### 2.3 权限模型（RBAC）

前端在最外层用 `AuthProvider` 解析 token，得到当前用户的 permissions 列表。

权限粒度设计：

```
admin                  → 超级管理员，可操作所有产品的 Admin 模块
admin:cowatch          → 只能操作 appKey = cowatch 的 Admin 模块
admin:video-to-audio   → 只能操作 appKey = video-to-audio 的 Admin 模块（未来）
can_view_monitor       → 可查看所有产品的 Monitor 模块
can_view_business      → 可查看所有产品的 Business 模块
```

后端 adminMiddleware 校验逻辑：

```
有 admin 权限         → 放行所有产品的 Admin 接口
有 admin:cowatch 权限 → 只放行 appKey === 'cowatch' 的 Admin 接口
否则                  → 403
```

前端路由守卫统一处理，三个模块完全同构，只换 `require` 字符串：

```tsx
<PermissionGuard require="can_view_monitor">
  <MonitorLayout />
</PermissionGuard>

<PermissionGuard require={`admin:${appKey}`} fallback="admin">
  <AdminLayout />
</PermissionGuard>
```

**前端 PermissionGuard 是体验保护，防止普通用户误操作。**
**后端 adminMiddleware 是安全边界，防止绕过前端直接调接口（curl 等）。**

两层都必须存在，缺一不可。

### 2.4 前端调用关系

```
cowatch-admin 前端
  ├── GET /monitor/stats/*         → monitor-backend（性能/错误统计）
  └── GET/POST /api/admin/*        → CoWatch-backend（业务数据 + Admin 操作）
```

不需要新建独立后端，直连两个现有服务即可。

### 3.1 路由隔离

Admin 接口统一使用 `/api/admin/*` 前缀，与现有业务路由物理隔离：

```
src/routes/
  auth/index.ts         ← 不变
  rooms/index.ts        ← 不变
  admin/
    index.ts            ← 统一注册 adminMiddleware，子路由自动继承

    rooms.ts
```

## 四、PostgreSQL 架构下的 Admin 设计

### 4.1 当前数据库架构

CoWatch-backend 已迁移到 PostgreSQL（`postgres.js` 连接池），数据库作为独立容器运行，与业务服务通过 Docker Compose 内部网络连接。

```
CoWatch Compose 网格
  ├── backend 容器（CoWatch-backend）── 网络连接 ──┐
  ├── postgres 容器（PostgreSQL 16）  ◄────────────┘
  └── frontend 容器（Nginx）

monitor-backend（独立部署，独立 SQLite 文件）
```

### 4.2 架构决策

PostgreSQL 是独立网络服务，天然支持多客户端并发连接：

- Admin 接口需要操作 CoWatch 数据库 → **Admin 接口写在 CoWatch-backend 里**（直连同一 PG 实例，无需聚合层）
- 监控数据查询接口 → **由 monitor-backend 自身提供（已有 /monitor/stats）**
- cowatch-admin 前端不直连数据库 → **通过 HTTP 调用两个现有后端**

数据库归属决定了代码归属；当前只有一个业务 backend，无需额外聚合层。

### 4.3 当前 Docker Compose 结构

```yaml
services:
  frontend:        # CoWatch 前端（Nginx）
  backend:         # CoWatch-backend（含 /api/admin/* 路由）
  postgres:        # PostgreSQL 16（backend 通过内部网络连接）
  # monitor-backend 独立部署，不在此 compose 文件中
```

cowatch-admin 作为独立静态站单独部署，不进入现有 compose。

### 4.4 未来多产品时的演进路径

当第二个子产品上线时，触发架构升级决策：

**路线 A：各产品共享 PostgreSQL 实例**

新产品 backend 直接连接同一 PG 实例（不同 database 或 schema 隔离）。改动最小，适合产品间数据有关联的场景。

**路线 B：新建 admin-backend 作为聚合层（推荐）**

```
admin-backend（不持有数据库）
  ├── → CoWatch-backend 内部接口
  ├── → monitor-backend 内部接口
  └── → future-product-backend 内部接口
```

admin-backend 只做请求转发和统一鉴权，各子产品 backend 保持独立。扩展性好，各产品可自由选型数据库。

**当前阶段（只有 CoWatch）不需要提前建 admin-backend，等第二个产品上线时再决策。**

## 五、多产品扩展性

三个模块均通过 `appKey` 隔离不同子产品数据：

```
/cowatch/monitor/perf           → appKey = cowatch
/cowatch/admin/users            → appKey = cowatch
/video-to-audio/monitor/perf   → appKey = video-to-audio（未来）
/video-to-audio/admin/users    → appKey = video-to-audio（未来）
```

新子产品接入时，前端只需：
1. 在 `src/pages/Business/` 下新建对应目录
2. 在 `src/api/` 下新增对应 backend 的 API 文件
3. 在路由配置里加一条 `/:appKey` 分支
4. 在首页产品列表里加一个入口卡片

完全是加法操作，不改动现有代码。

> **后续可考虑**：增加 `appKey = 'all'` 的跨产品聚合视图，用于对比各子产品的总用户量、总流量等全局指标。当前阶段目标是各子产品独立监控，暂不实现。

## 六、部署网络拓扑与端口分配

### 6.1 完整端口一览

```
宿主机对外（安全组开放 80、443 即可）：
  80 / 443       → 宿主机 Nginx（SSL 终止 + 子域名分发）
  6000           → daibao-dashboard 生产实例
  6001           → daibao-dashboard 测试实例

宿主机内部（仅本机可达，127.0.0.1 绑定，安全组无需开规则）：
  8000           → cowatch-backend 生产容器（容器内 3002）
  8001           → cowatch-backend 测试容器（容器内 3002）

其他独立部署（不属于 cowatch compose）：
  3070           → cowatch-frontend 生产
  3071           → cowatch-frontend 测试
  3100           → monitor-backend 生产
  3101           → monitor-backend 测试
```

**端口段约定（便于记忆和扩展）：**

| 段 | 用途 |
|----|------|
| `3070~3079` | CoWatch 前端对外入口（生产 3070，测试 3071） |
| `3090~3099` | video-to-audio 系列 |
| `3080~3089` | carry-hub 系列 |
| `3100~3109` | monitor-backend（3100 生产，3101 测试） |
| `6000~6009` | daibao-dashboard 对外入口（6000 生产，6001 测试） |
| `8000~8009` | 仅宿主机本地可达的后端内部端口（8000 cowatch 生产，8001 cowatch 测试） |
| 容器内统一 | `3002`（backend）/ `80`（nginx） |

### 6.2 两段 Nginx 代理职责分工

daibao-dashboard 请求经过两段 Nginx 处理，各自职责完全不同：

```
用户浏览器
    ↓  HTTPS
【宿主机 Nginx】（/etc/nginx/conf.d/daibao.conf，无此条目，daibao 不绑域名）
    ↓  由用户直接访问宿主机 6000/6001 端口
【容器内 Nginx】（daibao-dashboard 镜像内置，监听 80）
  职责一：静态资源服务（React 产物，SPA 兜底 /index.html）
  职责二：/api/*        → proxy_pass http://host-gateway:8000（cowatch-backend）
  职责三：/monitor-api/* → proxy_pass http://host-gateway:3100（monitor-backend）
    ↓
【cowatch-backend / monitor-backend】（宿主机上其他 Compose 的容器）
```

**容器内 Nginx 的 URL 如何注入：** `nginx.conf` 中使用 `${COWATCH_BACKEND_URL}`、`${MONITOR_BACKEND_URL}` 占位符，镜像启动时通过 `envsubst` 将 Docker Compose `environment` 注入的值替换为实际 URL，无需重新构建镜像即可切换生产/测试指向。

### 6.3 端口绑定：`0.0.0.0` + 安全组双重保护

```yaml
# cowatch/docker-compose.prod.yml
backend:
  ports:
    - "8000:3002"  # 0.0.0.0 绑定；安全组未开放此端口，外网不可直连
```

cowatch-backend 的 8000 端口使用 `0.0.0.0` 绑定（省略 IP 前缀的默认行为），而非 `127.0.0.1`。原因：daibao 容器经 `host-gateway` 访问宿主机时走的是 Docker 网桥 IP（如 `172.17.0.1`），不经过 loopback 接口，`127.0.0.1` 绑定的端口对容器不可达。

**安全性靠安全组保证**：云服务器安全组不开放 8000/8001 端口，外网无法直连，无需 `127.0.0.1` 绑定。daibao-dashboard 的 6000/6001 同样 `0.0.0.0` 绑定，供内网/VPN 访问。

### 6.4 跨 Compose 网格：host-gateway 机制

daibao-dashboard 与 cowatch 各自拥有独立的 Docker Compose 网格，容器间无法通过服务名（如 `http://backend:3002`）互访。

解决方案：在 `docker-compose.yml` 中声明 `extra_hosts`：

```yaml
daibao:
  extra_hosts:
    - "host-gateway:host-gateway"  # Linux 下由 Docker 自动解析为宿主机 IP
```

容器内 `/etc/hosts` 会增加一条 `host-gateway → 172.17.0.1`（实际 IP 取决于 Docker 网桥），容器内 Nginx 用 `http://host-gateway:8000` 即可穿透 Compose 边界，访问宿主机上由 cowatch compose 暴露的 `127.0.0.1:8000`。

### 6.5 当前耦合与未来解耦路径

**当前：** daibao-dashboard 直接依赖 cowatch-backend（通过 `host-gateway:8000` 跨 Compose 网格访问），没有独立的 admin-backend 聚合层。当前只有 CoWatch 一个产品，这是合理的简单方案。

**未来解耦（触发条件：第二个子产品上线）：**

新建 `admin-backend` 聚合层（见第四节 4.4），统一代理各子产品 backend 接口。

解耦后，daibao-dashboard 只需将 `COWATCH_BACKEND_URL` 从 `host-gateway:8000` 改为 `admin-backend:PORT`，nginx.conf 无需改动，容器内端口（80）和对外端口（6000/6001）均保持不变。
